Section references (§2, §3(x)) resolve to [m1-cooperative-awareness.md](../m1-cooperative-awareness.md); §5 (future-features register) now lives at [future/m1-future-features-register.md](../future/m1-future-features-register.md).

- **F1 — Week-2 hardware gate.** Commit to the UDP stub as the *M1 acceptance path*; real modem/PC5 work (R6 hardware KPI) proceeds only if an OBU is in hand by end of week 2 (A); otherwise R6 runs its mock-provider KPI and PC5 stays deferred (plan §6 allows this). Prevents the comms track from stalling convergence. Also note: vendor PC5 SDKs are proprietary — any use is a user-authorized hardware exception to the open-source constraint, decided by the user, never silently absorbed. **Closed 2026-07-08:** first resolved by the environment (no modem/baseband on the platform — FPT-Mentor: radio out of scope), then by user decision **hardware is removed from the entire project**, not just M1; [m1-product-fit-quectel-modem.md](../m1-product-fit-quectel-modem.md) is reference-only, and 3rd-party simulator candidates are researched in [m1-v2x-simulator-tools.md](../m1-v2x-simulator-tools.md).
- **F2 — Demo-dataset checklist trigger (R18).** The plan says video is "provided". The CARLA contingency task (~1 week, GPU machine) fires at week 3 unless the provided package contains **all four**: (i) two synchronized clips (A-view, B-view) from one convoy; (ii) C occluded from A for the demo duration; (iii) per-frame ground-truth d_AB/d_BC/d_AC with declared reference-point convention; (iv) intrinsics for both cameras. A video that exists but fails the checklist still fires the contingency — otherwise R11/R15 are untestable.
- **F3 — GNSS→chrony/NTP timebase fallback.** Without modem GNSS, R5's timebase is chrony/NTP at ≤ 50 ms (A). This satisfies "common timebase" for the demo; GNSS-disciplined (≤ 10 ms) is out of scope with hardware removed from the project (2026-07-08). Precision-of-requirement change, not scope change. **Env 2026-07-08:** satisfied by construction — platform nodes share the host clock; chrony still measures per R5's KPI.
- **F4 — AGPL-3.0 detector: accept or swap.** Ultralytics YOLO is open source (AGPL-3.0) and the C1/C2 pick, but virally binds the repo if distributed. Accept (project releasable under AGPL-3.0) or reject → pick switches to **YOLOX-s (Apache-2.0)**, ~2-day contained swap behind the detector interface.
- **F5 — KITTI is CC BY-NC-SA.** Used only as evaluation data — an instrument, never a shipped or linked component. Accept that reading, or reject → CARLA becomes the validation source too (folds into F2's task).
- **F6 — Strike the Adaptive AUTOSAR EM branch** of Phase 1: no open-source Adaptive EM exists, so the branch is unreachable under the hard constraints; systemd is the M1 mechanism. Plan-text cleanup, not a capability loss.
- **F7 — All (A)-marked numbers are proposals**, none verbatim in the plan: 10 Hz ± 10% rate; ≥ 99% delivery / 10-min soak; ≤ 50 ms NTP / ≤ 10 ms GNSS offset; 30 s boot; 120 s GNSS fix; 1 Hz GNSS log; ≥ 10 Hz / ≤ 100 ms perception; recall ≥ 90% (≤ 50 m) and lead ≥ 95% / 200 frames; ±15% per-leg (10–50 m, ≥ 90% frames); σ = 2 m flicker sweep; 300 ms p95 (200 ms target) E2E; 0.15·d_AC + 2.0 m ≥ 90% composed; 2 s ghost / 500 ms marker; week-2 and week-3 gates. **Round-2 additions:** ≥ 2 buttons; 300 ms view switch; 100 ms button response; ≥ 15 FPS @ ≥ 800×480 display; 30 s GUI auto-start; 50/100 ms p95 R20 latency; ≥ 99.9% R20 delivery / 10-min soak; week-4 render gate. Confirm or veto any individually.
- **F8 — Confirm CPM (ETSI TS 103 324), not SDSM (SAE J3224), as the schema's normative base.** ETSI's spec and ASN.1 are freely accessible → conformance auditable at zero cost under open-source-only; SAE is paywalled → unauditable. SDSM stays an informative reference.
- **F9 — IVI render target. CLOSED (round 2, 2026-07-08):** user decision D2 sets the render target to a **real GUI app on the Linux IVI vECU** (§3(i) pick); the AAOS branch stops being a rewrite risk because the same Dart codebase builds for Android/AAOS if the blueprint ever flips. Original fork (Linux-vECU OpenCV vs AAOS BEV app) preserved in §3(f)'s superseded text. Successor risk carriers: F13 (surface delivery), F12 (3D scope).
- **F10 — Slint GPLv3 on the IVI app (pre-decide NOW).** On an IVI/embedded target Slint is usable **only** under GPLv3 — the royalty-free tier explicitly excludes embedded systems and the commercial tier is paid ⇒ disqualified ([cache](../../.claude/references/slint-licensing-triple-license.md)). GPLv3 passes the open-source hard constraint exactly as AGPL did in F4, but virally binds the GUI app if distributed. **Accept** (GUI app GPL-licensed, fallback executable) or **reject** (Slint fallback off the table; if the week-4 gate then fails, the display track re-plans). This decision must precede the week-4 gate — an undecided F10 makes the fallback theater.
- **F11 — GPU-stack openness, deferred to the live-feed milestone.** CUDA/TensorRT are gratis but proprietary; the hard constraint bans commercial/*paid* tools, so they pass the letter — per the F4 precedent the closed-source nature is the user's call. Open alternatives named: ROCm (open-source), OpenVINO (Apache-2.0), ONNX Runtime (MIT). M1 carries zero GPU dependency (R10 round-2 note); the one-function detector interface (§3(d)) is the swap seam. Decision owner: user, at the live-feed milestone ([future-features register](../future/m1-future-features-register.md)).
- **F12 — M1 3D scope: 2D committed, 3D stretch.** User decision D6 says "2D or 3D", so acceptance = the 2D 3-car view; 3D is a best-effort stretch behind R19's view-interface seam (Flutter: perspective-`Transform` 2.5D on stable channel; true 3D awaits flutter_gpu or the Texture + native-GL hatch). Making 3D acceptance-gating would flip R19 to AT-RISK — flagged so the user can consciously accept or override.
- **F13 — GUI surface delivery on the cloud vECU + week-4 render-spike gate.** How judges *see* the IVI GUI (Wayland/weston headless-VNC, Xvfb+VNC, or a platform-provided viewer) is unconfirmed — confirm with FPT-Mentor at blueprint submission. The **week-4 render spike** on the actual vECU is the falsifiable gate for §3(i)'s pick: both Flutter runners tested, pass = ≥ 15 FPS @ ≥ 800×480 + input events; if neither passes, the Slint fallback fires (F10 must already be decided). Owner: user + [[project-architecture]] with the blueprint.
- **SC-1 — Accepted scope change (user-directed, recorded, already decided).** Plan §5 Phase 6 deferred a "production GUI later"; user decisions D2/D6 pull a production-shaped GUI (R19) into M1. Priced in the §2 round-2 feasibility note (~1 extra display-track week). Recorded per constitution rule 3 — a scope change is flagged even when the user directs it.
