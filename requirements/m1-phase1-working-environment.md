## The Carsky working environment

- **Platform:** cloud-hosted node platform, most likely AWS (team information, unconfirmed). The team calls it "Carsky"; FPT-Mentor's note says "CabinSky" — treated as the same platform family, naming unresolved and recorded as-is.
- **Scope framing (FPT-Mentor, authoritative):** this is an SDV hackathon — judged on software and the new in-vehicle experience. Simulating the 3GPP radio layer (PC5 PHY/MAC) is out of scope for both the environment and the grading criteria; invest no effort there.
- **Project scope corollary:** the team builds the *application* that sends/receives and consumes already-decoded V2X payloads — not a V2X protocol stack. Libraries are evaluated for codec/message-level use only (§ Suggested tools).
- **Environment model:** 1 node = 1 ECU; 1 vehicle = 1 blueprint (a topology of nodes). The team builds exactly one **ego** vehicle from its blueprint.
- **Outside world:** a single **bench node** runs a team-written **scenario player** that emits the V2X messages of N simulated vehicles into the ego — the test-equipment role, injecting data from outside (§ Phase 1 work targets).
- **No FPT-Mentor-provided V2X stack:** teams bring open-source libraries; everything runs userspace on ordinary sockets, with no special environment dependency. Decode placement: simplest inside the app on the IVI; more SDV-canonical is an extra Linux TCU node that decodes and forwards to the IVI.

### GNSS feed patterns (as described by FPT-Mentor; pending confirmation)

- **AAOS IVI node (virtualized Android):** the guest GNSS HAL receives data from the host over virtio/vsock (Cuttlefish/Trout pattern). Ego trajectory comes from the scenario player — synchronized with the simulated vehicles by construction — and flows GNSS HAL → LocationManager, so every guest app (maps included) sees it as real GPS; no app changes, no mock-location toggle.
- **Linux node (e.g. TCU/gateway):** the scenario player writes NMEA to a pty/TCP endpoint, gpsd attaches, apps read via libgps — or, simpler, the app reads a direct position feed when no real GNSS stack is exercised (recommended primary, [control/data-plane §5c](m1-phase1-control-data-plane.md)).

## Definitions — node, ego, blueprint

- **Node** — one ECU, virtualized on the platform. In the single-ECU case, one node hosts *all* modules of the component diagram (`V2X_Comm`, `ADA`, `IVI`).
- **Blueprint** — the ego vehicle's topology: which ECU nodes it comprises and how they link. With multiple ECUs, the diagram's modules split across nodes — e.g. a Linux TCU node running `V2X_Comm` decode, an AAOS IVI node running the display app.
- **Ego** — the one vehicle the team builds per its blueprint. All other vehicles (B, C) exist only as V2X message streams emitted by the bench node.
- **Bench node** — the platform node playing the outside world; runs the scenario player; not part of the ego blueprint.

## Phase 1 work targets

### Deliverable 1 — Scenario player (bench node)

Team-written program on the bench node; the world simulator in the test-equipment role. It must:

- Simulate the outside world: advance the trajectories of N simulated vehicles (M1: B and C) per a scripted, externalized scenario.
- Build and emit their V2X messages toward the ego — CAM (B's state) + CPM (B's perceived object C) — so the ego receives V2X traffic as if real peers existed.
- Generate message-level timestamps (`generationDeltaTime`) and the simulated vehicles' StationIDs; the ego's StationID stays in ego config.
- Optionally apply the configured impairment profile — drop/delay/jitter with a reproducible seed (FPT-Mentor-recommended option (b)).
- Feed the ego trajectory/GNSS (direct position feed primary; NMEA/gpsd or AAOS-HAL variants per § GNSS feed patterns, pending confirmation).
- Encode with asn1tools against the official ETSI ASN.1 modules (§ Suggested tools); bench-side control plane is plain process control — scenario start/stop, per [control/data-plane §2](m1-phase1-control-data-plane.md).

Requirements served: R3 (peer side), R4 (peer mirror of the parse log), R5 (timebase by construction), R18 (scenario script as a ground-truth source).

### Deliverable 2 — Ego modules, ECU node(s), blueprint

- **Module set** (component diagram): `V2X_Comm` — transport adapter + UPER decoder (R3, R4); `ADA` — `GnssProvider`, TrackedObject environment model, Phase-1-mocked situation assessment (R8, R12, R14, R15); `IVI` — **R19 GUI shell app** (buttons + central view window; framework per [report §3(i)](m1-cooperative-awareness.md)) whose central view renders the R16 ghost-C 3-car scene, fed by the R20 ADA→IVI data path (mocked seam Phase 6 consumes) — *round-2 update 2026-07-08: replaces the earlier "2D BEV occluded-vehicle warning" wording (R16, R19, R20)*.
- **ECU placement options:** (a) one Linux vECU node hosting all three modules — simplest, matches FPT-Mentor's decode-in-app suggestion; (b) Linux TCU node (`V2X_Comm`) + IVI node (display) — more SDV-canonical, adds one inter-node link. The placement decision belongs to [[project-architecture]]. *(Round-2 note: R20's UDP data path works identically in both placements — the choice stays open.)*
- **Blueprint + use case for FPT-Mentor:** draft the ego blueprint (which ECU nodes; what runs real vs receives data from the bench node) and the use-case description (NLOS occluded-vehicle warning: A is warned about C, relayed by B) — FPT-Mentor's starter-pack, resource, and transport advice is gated on receiving both. FPT-Mentor's own example driver-facing use cases, given for calibration: blind-intersection occluded-vehicle warning, priority-vehicle-approaching alert, hazard-ahead warning, green-light-optimal-speed advisory (GLOSA). M1's NLOS occluded-vehicle warning matches the first example directly. *(Round-2 note: the blueprint submission should also raise flag F13 — how the IVI GUI surface is displayed/streamed to judges.)*
